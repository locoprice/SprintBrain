F1 High Dashboard never sets lang_group_id — cross-platform language grouping is silently broken

app/src/lib/api/snippetsApi.ts:205 (createSnippet insert) & :334 (duplicateSnippet) · consumed by extension/background/background.js:102 & extension/content/content.js:428 (_findLangVariants)

The extension's whole multi-language UX (the "🌐 Available in multiple languages" picker, the EN/ES/IT variant collapsing in the context menu and pickers) is driven primarily by the lang_group_id column. The extension queries it and _findLangVariants() uses it as the first-choice grouping key. But the dashboard's createSnippet / duplicateSnippet insert payloads omit lang_group_id entirely — the column never appears in SNIPPET_SELECT or in any insert/update path.

Result: every snippet authored in the dashboard has a null lang_group_id and can only be grouped via the fragile fallback heuristic (strip a trailing EN/ES/IT/FR/MULTI from the shortcut). Two variants whose shortcuts don't share a stripped base — e.g. /welcome (EN) and /bienvenida (ES) — will never be detected as siblings: the language picker won't appear and they surface as two unrelated rows. This is a core advertised feature degrading silently, and it only manifests on the extension, so dashboard-only testing misses it.

Evidence
// snippetsApi.ts — insert payload has no lang_group_id
const insert = { id, user_id, title, shortcut, body, bodies, lang,
  folder_id, field_cfg, sort_order, ... };  // ← lang_group_id missing

// content.js _findLangVariants — primary key is lang_group_id, then heuristic
if (item.lang_group_id) { ...group by id... }
if (Object.keys(map).length <= 1) { ...fallback: shortcut.replace(LANG_SUFFIX_RE,'') }
Recommended: Make the dashboard a first-class writer of lang_group_id: add it to SNIPPET_SELECT, the create/duplicate inserts, and the edit path. Generate a shared group id when the user links variants (and reuse the source's group id on duplicate). Add a vitest case asserting created sibling variants share a group id — the existing suite mocks the API so this gap is invisible to tests today.

F2 Medium Formula divide-by-zero outputs literal "Infinity" into customer-facing messages

extension/formula-engine.js:215 (evalFormula) · operator at :161

safeEval performs real JS division, so 1/0 yields Infinity. The guard in evalFormula rejects only null and NaN — not non-finite numbers — so Infinity flows straight through Math.round() and is stringified into the inserted text. Verified live:

$ node -e 'FE.resolveBody("Price: {{= 1/0 }}", {})'
→ "Price: Infinity"

For a hospitality quoting tool whose flagship snippets compute prices ([YOUR_PRICE × 1.03], [OTA_PRICE - YOUR_PRICE]), a stray Infinity (or, with a slightly different guard order, -Infinity) reaching a real guest message is a quality/trust defect in the most important path. Empty/garbage operands already coerce to 0, so this is the one remaining hole.

Recommended: Change the guard at line 215 to reject non-finite results: 
return (r === null || !isFinite(r)) ? null : Math.round(r * 100) / 100;
Returning null makes resolveBody emit "" (its existing behavior for unresolved formulas), which is far safer than "Infinity". Add a formulaConditions test case for x/0.

F3 Medium No code-splitting — one 1.2 MB JS chunk on first load

app/ build output · app/src/App.tsx (all routes imported eagerly)

vite build emits a single index-*.js at 1,201 KB (327 KB gzip), tripping Vite's 500 KB advisory. Everything — Recharts, the Supabase client, every route — ships in one bundle even though the analytics charts (Recharts is heavy) are only needed on one page. The project's own root standard names "extreme ease of navigation / speed" as the primary goal and asks to "watch bundle growth and Core Web Vitals," so this is on-mission, not cosmetic.

Recommended: Lazy-load routes with React.lazy + <Suspense> in App.tsx (Analytics especially), and add build.rollupOptions.output.manualChunks to split out recharts and @supabase/supabase-js. This typically cuts the initial payload by well over half with no behavior change.

F4 Medium Docs claim signups are gated to @leibtour.com — the gate was removed

app/CLAUDE.md §1 & §4.8 vs services/supabase/migrations/20260518120000_drop_auth_domain_allowlist.sql

app/CLAUDE.md states auth is "domain-restricted to @leibtour.com" and that "Only @leibtour.com emails can sign up (enforced by a BEFORE INSERT trigger on auth.users)." That trigger and its function were dropped in May 2026 — the migration comment explicitly says "SprintBrain is now an open SaaS product that accepts any valid email address," and SignupPage.tsx enforces no domain. The docs are the canonical onboarding reference (root CLAUDE.md mandates reading them before changes), so a developer reasoning about access control from the docs would draw a false, security-relevant conclusion (e.g. assume an allowlist exists and skip a check). The behavior is intentional; the documentation is stale.

Recommended: Update app/CLAUDE.md §1 and §4.8 to describe the open-signup reality and reference the drop migration, so the only access-control statement of record matches the database.

F5 Low Rate limiter is in-memory & client-side only

app/src/lib/rateLimiter.ts · used in SignupPage.tsx:201, AuthModal.tsx:589/645/676

checkRateLimit counts attempts in a module-level Map. It resets on every page reload, is per-tab, and is bypassed entirely by anyone calling Supabase directly. It guards OTP send, password sign-in, and recovery — exactly the endpoints an attacker would target. Real protection here comes from Supabase's server-side limits (the function correctly handles HTTP 429 in extension/auth/auth.js), which makes the client limiter a UX nicety, not a control. The risk is mainly that it reads as a security measure.

Recommended: Keep it as UX debounce but document it as non-authoritative, and confirm Supabase Auth rate limits are configured server-side (they are the real boundary). No code change strictly required.

F6 Low Background message listener returns true for every message

extension/background/background.js:469-475

This onMessage listener handles only REFRESH_MENUS but return true runs for all messages, telling Chrome "I will respond asynchronously." For every other message type (e.g. log_event, auth_changed, handled by the other two listeners) it keeps a response channel open that it never closes, producing intermittent Unchecked runtime.lastError: The message port closed before a response was received noise and a small leak. The pre-commit standard explicitly forbids committing service-worker console errors.

Recommended: Only keep the channel open for messages this listener actually answers: 
if (msg && msg.type === 'REFRESH_MENUS') { initMenus(); sendResponse({ ok: true }); return true; }
return false;

F7 Low Intent-engine "Layer 2" LLM is a non-functional stub

app/src/lib/intentEngine.ts:233-249

When Layer-1 confidence falls below LLM_THRESHOLD (0.6), classifyPrompt calls classifyWithLLM, which is a stub that returns the same result with confidence cosmetically raised to ≥0.72 — no model is consulted. Any UI that displays this confidence, or treats ≥0.72 as "the LLM verified it," is showing a fabricated number for genuinely ambiguous prompts. This is clearly marked as a stub in-code, so it's a roadmap gap rather than a defect — flagged because the confidence bump is misleading if surfaced to users.

Recommended: Until the Claude call is wired, have the stub return Layer-1 confidence unchanged (don't inflate it), or tag the result so the UI can label it "heuristic" vs "verified." When implementing for real, use the latest model (claude-opus-4-8).

F8 Low Bulk import runs sequential round-trips with no retry

app/src/stores/snippetStore.ts:452-473 (importSnippets)

Import loops await snippetsApi.createSnippet(item) one row at a time, so importing N snippets means N serial network requests (slow and visibly janky for large libraries — the production workspace already has ~158 rows). Failures are counted but not retried or surfaced per-row.

Recommended: Batch with a single multi-row insert (mirroring bulkMoveSnippets/bulkDeleteSnippets), or bound concurrency (e.g. Promise.all over chunks). Return the failed items so the UI can report which rows didn't import.

F9 Info Documentation drift

docs/CLAUDE.md · AUDIT_SUMMARY.md · app/CLAUDE.md §2

Several docs lag the code: docs/CLAUDE.md reportedly still says "Testing: Manual only / no test framework" while a 10-spec, 148-test vitest suite is wired to npm run test (this was already F1 in the prior AUDIT_SUMMARY.md and remains open); AUDIT_SUMMARY.md's header still reads v2.63.8 while the repo is at v2.63.10; and the intent engine's preferred-model table points at claude-opus-4-7/sonnet-4-6 (one generation behind the current claude-opus-4-8). Individually trivial, collectively they erode the "docs are source of truth" guarantee the repo relies on.

Recommended: Refresh the testing section, bump the audit header, and revisit the model map when convenient.

F10 Info Dead write-only accumulators in the content script

extension/content/content.js:1450-1451, 1542

totalSecs and totalSnips are declared and incremented on every celebration confirm but are never read anywhere in the bundle — dead state. The root standard forbids dead code in production. (Harmless, but it's exactly the kind of leftover the zero-defect bar targets.)

Recommended: Remove both, or actually surface the "seconds saved" lifetime stat in the popup if it was intended to be displayed.
